# 青砚 · AI 虚拟团队协作手册

本文档定义**五个虚拟角色**的职责边界、产出物与协作顺序，供你在 Cursor 中**分会话 / 分任务**调用（每次对话可声明「你现在扮演 XXX」并粘贴对应段落）。  
目标：在**不混淆职责**的前提下，完成**大型重构与 UI/UX 升级**（全中文产品）。

---

## 团队角色一览

| 角色 | 英文代号 | 核心使命 | 主要产出 |
|------|-----------|----------|----------|
| 产品经理 | **PM** | 想清楚「做什么、先做什么、什么叫做完」 | PRD 片段、优先级列表、验收标准 |
| 架构师 | **ARCH** | 技术边界、数据与 API 一致性、可演进性 | 架构说明、模块图、技术债清单、重构分期 |
| UI/UX 设计师 | **DESIGN** | 信息架构、视觉与交互一致性、中文体验 | 页面层级、组件规范、关键流程线框级说明 |
| 全栈开发 | **DEV** | 按架构与设计落地代码 | PR、迁移脚本、实现说明 |
| 测试 / 质量 | **QA** | 可重复验证、回归范围、上线门禁 | 测试清单、边界用例、发布前 Checklist |

**协作顺序（推荐）**  
`PM 定范围与优先级` → `ARCH 定拆分与约束` → `DESIGN 定界面与交互原则` → `DEV 实现` → `QA 验证` → `PM 验收`

并行规则：**DESIGN 可与 ARCH 草案并行**，但**大规模改 UI 前**需 ARCH 确认路由与数据流不会推翻设计假设。

---

## 各角色：职责 / 输入 / 输出 / Cursor 用法

### 1. PM（产品经理）

**职责**

- 明确用户场景与 MVP 边界；抑制范围蔓延。
- 将「朋友遗留的不清晰需求」整理为**可验收**条目。
- 维护 **P0 / P1 / P2** 与里程碑（例如：稳定性 → 核心工作流 → 行业模块）。

**输入**  
`docs/PROJECT_AUDIT.md`、现有 README、侧栏真实入口、你方业务目标（出口厂家等）。

**输出（建议模板）**

- 本期目标一句话 + 非目标（明确不做）。
- 用户故事列表：`作为…我希望…以便…`
- 验收标准：每条功能 **Given / When / Then**。

**在 Cursor 中开启对话时粘贴**

```text
你现在是青砚项目的 PM。只讨论需求范围、优先级和验收标准，不写具体代码。
请先阅读 docs/PROJECT_AUDIT.md，基于事实列出 P0/P1/P2，并指出与 README 不一致之处。
```

---

### 2. ARCH（架构师）

**职责**

- 对照 **Prisma schema 与真实页面**，标出「模型很大但产品未暴露」的落差。
- 划分模块：**认证与会话、任务/项目、日历与提醒、工艺单（百叶窗）、组织/多环境、Prompt 与知识库** 等。
- 定义 API 与前端调用的**契约**（错误码、分页、权限假设）。
- 提出**分期重构**：哪些可独立拆包、哪些必须一次迁移。

**输入**  
`prisma/schema.prisma`、`src/app/api/**`、`middleware.ts`。

**输出**

- 一张逻辑模块图（可用 Mermaid）。
- 「技术债 TOP 10」+ 每条风险等级。
- 重构阶段建议（例如：Phase A 仅 UI 壳 + 设计系统；Phase B 任务域模型收敛）。

**在 Cursor 中开启对话时粘贴**

```text
你现在是青砚项目的架构师。请阅读 prisma/schema.prisma 与 src/app 下的路由，
输出模块边界说明、与 README 描述不符之处、以及分三期重构的建议（每期可独立上线）。
不要改 UI 视觉细节，除非涉及路由或信息架构。
```

---

### 3. DESIGN（UI/UX 设计师）

**职责**

- **全中文**：文案语气、字号层级、留白、表单错误提示、空状态。
- 建立**设计系统雏形**：主色/中性色/语义色、圆角、间距阶梯、按钮与表格密度。
- 梳理**关键路径**：登录注册 → 工作台 → 任务/项目 → AI 助手 → 设置（含第三方连接）。
- 标注**当前 UI 不一致点**（同类页面组件不统一、移动端未考虑等）。

**输入**  
`(main)/layout.tsx`、`components/sidebar.tsx`、`globals.css`、典型页面如 `tasks`、`projects`、`assistant`；**规格真源**：[DESIGN_SPEC_PHASE1.md](./DESIGN_SPEC_PHASE1.md)。

**输出**

- 全局布局与导航的信息架构建议。
- 「组件级规范」列表（Button / Card / Table / Modal / Form）。
- 若需大改：给出**分页面改造顺序**（先壳后页）。

**在 Cursor 中开启对话时粘贴**

```text
你现在是青砚项目的 UI/UX 设计师，产品语言为简体中文。
请先阅读 docs/DESIGN_SPEC_PHASE1.md，在其 Token/字阶/五组件框架内做迭代或补充 Figma；
输出要可交给全栈直接实现，避免抽象形容词堆砌。
```

---

### 4. DEV（全栈开发）

**职责**

- 在 ARCH 约束下实现；在 DESIGN 规范下改 UI。
- Next.js App Router、Server/Client 边界、Prisma 迁移策略与回滚注意点。
- 小步提交：每个 PR **一个主题**（例如仅「设置页」或仅「任务列表」）。

**输入**  
PM 验收标准 + ARCH 模块边界 + DESIGN 规范。

**输出**

- 代码与必要迁移；**不扩大范围**。
- 简短「变更说明」：改了什么、如何验证。

**在 Cursor 中开启对话时粘贴**

```text
你现在是青砚项目的全栈开发。请严格按 docs/AI_TEAM.md 中的分工，
在本次任务中只实现 [具体范围]，遵循现有 TypeScript 与 Tailwind 风格；
修改前先用 @ 引用相关文件，避免无关重构。
```

---

### 5. QA（测试 / 质量）

**职责**

- 基于 PM 的验收标准写**可执行**测试清单（手工为主，后续可加自动化建议）。
- 覆盖：认证 Cookie、AI 流式、Google OAuth、日历同步、工艺单计算与导出、多标签页会话等。
- **回归范围**：每次发布列出必测路径。

**输入**  
本次 DEV 的变更说明、环境变量清单（`.env.example`）。

**输出**

- 用例表：步骤 | 预期 | 实际（留空给你填）。
- 发布前 Checklist（Vercel 环境变量、迁移、OAuth 重定向）。

**在 Cursor 中开启对话时粘贴**

```text
你现在是青砚项目的 QA。请根据 docs/PROJECT_AUDIT.md 中的功能面，
输出一份 P0 手工回归清单（含认证、任务 CRUD、AI 助手、Google 日历、设置），
并标出依赖环境变量的用例。
```

## 建议的「第二期：整体优化」执行节奏

1. **第 0 周（对齐）**  
   PM + ARCH 更新 `PROJECT_AUDIT.md` 中的「差距与决策」；冻结一期范围。

2. **第 1 波（体验底座）**  
   DESIGN + DEV：`globals.css` / 布局 / 侧栏与顶栏 / 通用组件；**不改业务规则**。

3. **第 2 波（核心工作流）**  
   工作台、任务、项目、收件箱：列表与详情交互统一；空状态与加载态。

4. **第 3 波（垂直模块）**  
   工艺单、组织、Prompt/知识库：按 PM 优先级决定是否「瘦身暴露」或「隐藏未就绪入口」。

5. **持续**  
   QA 每波结束跑 P0 清单；ARCH 审查是否引入新耦合。

---

## 与你个人目标的对应

- **二手项目不熟**：由 **PM + ARCH + 本审计文档** 先把「代码里到底有什么」对齐到文档。  
- **要大改 UI/UX**：由 **DESIGN 定规范 → DEV 分波落地 → QA 守门**。  
- **要最好效果**：避免「一个对话里同时当五个人」；**分角色、分批次** 推进。
